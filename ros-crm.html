ROSnet Operating Model

Sales · Finance · CRM

Sales overview

Pipeline, rep workflow, deal rules, tools, gaps, and decisions to define.

Purpose

Manage qualified commercial opportunities without inflating pipeline or creating duplicate companies.

Core rule

Deal = concept being sold. Company = legal operator. New location of existing concept is not a Sales deal.

Main risks

MRR vs ARR confusion, inconsistent naming, deleted ghosted stage, and missing prioritization views.

Sales — questions to define

New concept for same legal entity: New Business or Expansion?

When an existing client adds a new concept, Sales needs a clear deal type rule. This affects pipeline categorization, expansion reporting, and forecast accuracy.

Deal distribution rule between sales reps

Inbound leads are assigned case by case today. Define whether distribution should be by geography, segment, concept type, ownership, or rotation.

ICP Score criteria

Define High / Medium / Low criteria before configuring prioritization views in HubSpot. Possible inputs: location count, concept type, ownership model, geography, and current tech stack.

Deal naming and searchable site name

Deal naming is [Operator] — [Concept], but reps also search by internal site name. Decide whether site name appears in the deal name or as a separate searchable field.

Finance overview

Billing setup, Maxio, first billing, location changes, monthly cycle, gaps, and decisions to define.

Purpose

Translate signed contracts, implementation readiness, locations, modules, and billing entities into accurate Maxio billing.

Core rule

Maxio subscription = Site. MRR lives at subscription level. Billing entity structure depends on client preference.

Main risks

Manual entry, site_id not searchable at customer level, and multiple quantity updates per module for location changes.

4 fully manual steps — every new client

Signed agreement → Implementation Tracker → Maxio customer → subscriptions per site. Each step is a point of failure: misspellings, missed steps, delays.

No automation, no validation, no integration. site_id not at customer level in Maxio — blocks any future sync.

Once activated: fully automatic

The only recurring manual step is the monthly confirmation with the Implementation Team to add new clients to the billing cycle. All existing clients bill automatically from Maxio.

This confirmation should be triggered automatically when a deal reaches Closed Won in HubSpot.

Two alerts · two manual updates · per module

If the client has Labor, API, and Power Center — that is 3 separate quantity updates in Maxio for every new location.

For a client with 5 modules opening 3 new stores: 15 manual updates. Maxio API could automate all of this with a single event trigger.

One close alert · multiple manual updates

Stayce must update each product module separately in Maxio. If the client has 3 modules — 3 separate quantity changes per location closure.

Target: close alert → Maxio API → all modules updated in one automated call.

Once activated: fully automatic

All existing clients bill automatically on the 1st. Stayce's only recurring manual work is confirming new clients each month with the Implementation Team.

This monthly confirmation should be triggered automatically by Closed Won in HubSpot — eliminating the last manual step.

Finance — questions to define

Fix site_id in Maxio — subscription level vs customer level

The rosnet_site_id currently exists at subscription level only. Finance and the tech team need to decide how to expose it at the customer / billing entity level so HubSpot ↔ Maxio sync can work.

Should Billing Sub-Entities exist in HubSpot?

Billing structure varies by client. Decide whether billing sub-entities remain Maxio-only or become visible in HubSpot for reps and CS.

Location tracking: count field or start/end dates?

Decide whether HubSpot only needs a location count per Site, or whether each location needs lifecycle dates for billing and reporting.

CRM overview

Account hierarchy, CS handoff, account ownership, activity logging, data architecture, gaps, and decisions to define.

Purpose

Make HubSpot the operational center for account history, ownership, hierarchy, renewals, and cross-system visibility.

Core rule

Company-level activity is the shared history. Sales can own deals while CS owns the account after Closed Won.

Main risks

Incorrect hierarchy, CS meetings contaminating Sales deals, broken rollups, missing contract renewal tracking, and unsupported SOW revenue.

—

Concept / Brand — not a hierarchy level

Wendy's, IHOP, Applebee's are brand labels — a field on the Site object, not a separate record or hierarchy level. One operator can have multiple concepts. Each concept gets its own Site and Deal, not its own Company.

Three real configurations — same model, different permutations

Simple operator

No holding group · 1 operator · 1 concept · 1 site

e.g. Lenz Enterprises LLC operating Cheba Hut — 7 locations. One Company, one Deal, one Site.

Multi-concept operator

Holding group → 1 operator · 2 concepts · 2 sites

e.g. Good Times Restaurants, Inc. operating Good Time Burgers (27 locs) + Bad Daddy's (37 locs). One Company, two Deals, two Sites.

Complex group

Enterprise → multiple legal entities · multiple concepts · multiple billing sub-entities

e.g. Hamra Enterprises → Hamra Chicago LLC + Wendy's of Missouri Inc. + Wendy's of New England LLC. Three Companies, N Sites, 4 billing sub-entities.
